    API Overview

    What is SmartInventory?

    The SmartInventory API lets you create inventory management applications that use IoT-enabled devices (i.e. barcode scanners and printers) to collect data from scan events in warehouses to track products as they move through stocking to shipping.

    Who is it for?

    Developers creating applications for warehouse management that track stock levels, forecast sales, identify gaps in quality control, and create audits. You can also issue commands to devices using our API to do things like print physical copies of crew reports in the warehouse or change settings in a scanner.

    Why use it?

    Our API manages the interaction between your customers' warehouse devices and your applications, so you do not need to reinvent the wheel. In the end, that means saving costs and cutting down on your development time so you can turn a profit sooner than later.

    How do I use it?

    After you have completed sign up and registeration, we provide you the credentials you need to request access to our API. You can then start to enroll your devices at a warehouse area, activate them for data capture, send them commands to perform certain actions, and request device statuses to ensure they are working. This doc site was creating to walk you through entire process of using our API!

    Get Started

    Authorization

    Our web-hosted resources are protected through the Microsoft identity platform that implements OAuth 2.0 authorization protocol. OAuth 2.0 is a method through which your third-party app can access web-hosted resources on behalf of your users.

    Before you integrate our IoT API, your application must establish authorization via the Microsoft identity platform's OAuth 2.0 authorization protocol to access our API.

    Set up account

    You need a developer, personal, or business account to log into our developer portal, gain credentials, and create an API test environment. Each account provides different levels of access to SmartInventory functionality.

    Capabilities Developer Personal Business
    Access test environment x x x
    Make API requests x x
    Go live x

    If you would like to access a capability you currently do not have, you may upgrade your account on your Account page in the Developer portal.

    Get credentials

    To generate API credentials for your test and live environments:

    1. Log into the Developer Portal with your SmartInventory account credentials. If you do not have an account, click Sign Up to set up an account.

    2. On My Web Apps & Credentials, use the tab to alternate between your Live and Test web apps.

    3. Enter a name for your app and click Create Web App.

      • Your credentials are displayed on the resulting page. You will need to insert these crendentials into the header of your requests.

    4. Review your app information and click Save.

    A note on request bodies

    There are no path parameters or query string parameters. You simply submit JSON to the API in the request body. The sample request bodies are provided for each end point. Also, the only method used with this API is POST.

    Use of Postman variables in headers

    For simplicity, Postman variables have been inserted into the code samples, indicating certain values need to be replaced by the ones provided to you during registeration.

    Postman variables are indicated by values wrapped in brackets in the parameter tables and code samples on the right pane, as the example shows below:

    {{bearer_token}}

    Also, Postman variables are only used in the request headers, and thus are provided in the Value column of the parameter tables to indicate the variable to be used for a given parameter. They are assumed to be mandatory.

    For the endpoints in the Resources sections, the parameters in the request bodies are indicated as Mandatory or Optional. There is no Value column since no variables are used.

    Request a bearer token to establish authorization to use our API. Enter your credentials in the header of your request, as shown below.

    HTTP Request

    POST /oauth2/token

    Header Parameters

    Param Value Type Description
    content_type application/x-www-
    form-urlencoded    		 string The format the body of the request is being sent as.
    resource_ID {{resource_ID}} string The fully qualified ID of the resource your application is requesting "access to".
    client_ID {{client_ID}} string The public identifier for your application represented as a 32-character hex string.
    client_secret {{client_secret}} string The secret (or password) known only to the application and the authorization server.

    Request Body

    Since the content_type for this request is application/x-www-form-urlencoded, there is no JSON example. Below is a request example:

    content_type=application/x-www-form-urlencoded&resource_ID={{resource_ID}}&client_ID={{client_ID}}&client_secret={{client_secret}}

    Param Value Type Description
    resource_ID {{resource_ID}} string The fully qualified ID of the resource your application is requesting "access to".
    client_ID {{client_ID}} string The public identifier for your application represented as a 32-character hex string.
    client_secret {{client_secret}} string The secret (or password) known only to the application and the authorization server.
    A successful request returns a 200 OK Status with the body returning the bearer token. OAuth2 is now established.

    Response

    Param Type O/M Description
    token_type string Mandatory The type of token (i.e. Bearer).
    expires_in datetime Mandatory The date and time the access token expires.
    resource_ID string Mandatory The fully qualified ID of the resource your application is requesting "access to".
    access_token string Mandatory (AKA "Bearer") token that informs the API that you are authorized to access the API and perform specific actions specified by the scope that was granted during authorization.

    Now that OAuth2 is established, use the bearer token to retrieve a session token. Please note that the bearer token expires, as indicated by the expires_in value returned in the response.

    Use a tool such as jwt.io to view the content of the bearer token. This approach is handy for troubleshooting if you need to verify permission roles granted for the client it is provided against.

    Enroll to the area encoded in your session token. The only required action is to include it in the request body. It is decoded and checked during the enroll call.

    HTTP Request

    POST area/enroll

    Request Body

    Param Type O/M Description
    session_token string Mandatory One-time token used to establish a session for a user and is revoked when the token expires. It contains the location at which the session token will be used.
    A successful request returns a status of 200 OK and the body of the response will contain the device details for the enrolled area.

    Response

    The response returns the device_details object that lists the devices at the specified location and their details such as their state, if they are activated, their type, and actions available to each device.

    Param Type O/M Description
    device_details array[object] Mandatory List of devices at the specified area and their details.
    device_state string Mandatory The overall state of the device, either: Online, Offline, or Error.
    is_activated bool Mandatory If yes, then the device has been activated by your application. If no, it is not activated.
    device_ID string Mandatory The network ID used to identify the device on the network.
    device_type string Mandatory The type of device, either Laser barcode scanner, RFID Scanner, or Direct Thermal Printer.
    device_actions array[object] Mandatory The actions available to the device at the location. Either read_2D_barcode, read_RFID_tag, or print_product_label.
    An unsuccessful request may return either a 400, 401, 409, 422 error code with an explanatory message. For the general meaning of these responses, see the Error Codes section of this document.
